# 代码质量评估技能设计文档

## 1. 系统概述

代码质量评估技能是一个**项目级代码质量评估框架**，用于对被评估项目的代码进行全面审查，按项目自身设计要求（README.md / AGENTS.md）进行评分，并生成标准化评估报告。

### 1.1 核心定位

| 属性 | 说明 |
|------|------|
| 类型 | AI Agent Skill（OpenCode 技能） |
| 触发方式 | 用户输入包含关键词：qa、代码质量、质量评估、代码评估、质量报告、代码审查、code quality、quality report |
| 评估标准 | 被评估项目**自己的设计文档**（非通用规范） |
| 输出产物 | 质量评估报告（Markdown 格式） |

### 1.2 项目结构

```
skills/qa/
├── SKILL.md              # 主技能文件（272行）- 核心执行指南
├── report.md             # 报告模板（120行）
├── checklist/            # 检查清单（结果填写模板）
│   ├── checklist.md       # 通用模板
│   ├── checklist_java.md  # Java 结果记录模板
│   └── checklist_nodejs.md # Node.js 结果记录模板
├── rules/                # 检查规则（定义检查项及定级）
│   ├── rules.md           # 评分规则 + 扣分标准
│   ├── rules_java.md      # Java 特有规则（270+检查项）
│   └── rules_nodejs.md    # Node.js 特有规则（270+检查项）
└── scripts/
    └── score_calculator.js # 评分计算器（命令行工具）
```

## 2. 核心设计原则

### 2.1 评估哲学

**"代码符合项目自己的设计要求 = 合格，违反 = 不合格"**

评估的标准是被评估项目自己的设计文档，不是通用规范。例如：
- 如果被评估项目的 README.md 定义了 DDD 分层规则，则违反即为 P0/P1
- 如果被评估项目的 AGENTS.md 禁止同步调用，则违反即为问题

### 2.2 强制执行规则

| 禁止事项 | 说明 |
|----------|------|
| ❌ 询问用户 | 不允许中断等待确认 |
| ❌ 提供选项 | 不允许让用户选择 |
| ❌ 跳过阶段 | 必须完整执行所有阶段 |
| ❌ 跳过代码阅读 | 必须逐行分析核心文件 |
| ❌ 跳过文档阅读 | README/AGENTS.md 必须读 |
| ❌ 跳过数字验证 | 必须交叉验证 |
| ❌ 随便编造问题 | 必须有代码证据（文件:行号） |
| ❌ 自行降级 | 必须按 rules + 设计文档定级 |
| ❌ 日志级别未区分 | 敏感信息泄露按日志级别定级 |

| 必须事项 | 说明 |
|----------|------|
| ✅ 完整执行所有阶段 | 不允许省略 |
| ✅ 保存报告到被评估项目根目录 | 不保存到 workspace |
| ✅ 每个问题标注文件和行号 | 无证据不报告 |
| ✅ 使用颜色标识问题等级 | P0🔴/P1🟠/P2🟡/P3🔵 |
| ✅ 问题描述基于实际代码 | 不基于猜测 |
| ✅ P0 问题必须流程走查 | 清单式检查无法发现 |
| ✅ 问题定级交叉引用 | rules + 被评估项目设计 |

## 3. 评估流程

### 3.1 四阶段执行流程

```
┌─────────────────────────────────────────────────────────────────┐
│  阶段 1: 扫描 + 读懂项目设计                                      │
│  ├── 扫描所有代码文件（跳过 node_modules/target/vendor）          │
│  ├── 读取 README.md（提取设计要求）                               │
│  ├── 读取 AGENTS.md/CLAUDE.md（提取分层/鉴权规则）               │
│  ├── 读取配置文件（路由/认证/日志级别）                            │
│  └── 识别项目概况（语言/框架/规模/架构）                           │
├─────────────────────────────────────────────────────────────────┤
│  阶段 2: 代码审查（双视角）                                       │
│  ├── 视角 A: 按被评估项目 README.md 设计要求检查                  │
│  │   └── 逐条对照设计要求与代码实现                               │
│  └── 视角 B: 通用缺陷模式检查（补充）                             │
│      └── 按 rules_{语言}.md 清单逐项检查                         │
├─────────────────────────────────────────────────────────────────┤
│  阶段 3: 业务流程走查（发现 P0 的关键）                           │
│  ├── API 入口走查（认证→鉴权→入参→调用链→异常→响应）             │
│  ├── 按设计要求逐条验证（设计 vs 代码行为）                       │
│  ├── 状态流转走查（画出状态机，验证非法跳转）                     │
│  ├── 并发安全走查（先查后改是否有并发保护）                       │
│  └── 敏感数据走查（密钥/Token/日志脱敏）                         │
├─────────────────────────────────────────────────────────────────┤
│  阶段 4: 问题合并 + 评分 + 验证 + 报告                           │
│  ├── 问题合并去重（同位置保留高定级）                             │
│  ├── 按维度计算扣分（维度扣分上限 = 权重分值）                    │
│  ├── 数字交叉验证（三轮验证）                                     │
│  ├── 按模板生成报告（6 部分）                                     │
│  └── 保存报告到被评估项目根目录                                   │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 阶段 1：扫描 + 读懂项目设计

#### 步骤 1.1：扫描所有文件

| 扫描范围 | 说明 |
|----------|------|
| ✅ 必须扫描 | 源码目录、配置文件 |
| ❌ 跳过 | 依赖目录（node_modules/target/vendor）、构建产物（dist/build）、IDE 配置、.git |

#### 步骤 1.2：读取项目设计文档

| 文档 | 提取内容 |
|------|----------|
| README.md | 项目简介、技术栈、架构设计、设计要求 |
| AGENTS.md / CLAUDE.md / GEMINI.md | 分层规则、鉴权规则、技术约束、禁止事项、优先级定义 |
| 配置文件（.yaml/.json/.env） | 路由规则、认证策略、日志级别、运营配置 |

#### 步骤 1.3：识别项目概况

| 属性 | 说明 |
|------|------|
| 项目名称 | {项目名称} |
| 开发语言 | {语言} |
| 技术框架 | {框架} |
| 架构模式 | {模式} |
| 代码规模 | {文件数} 个文件，约 {行数} 行 |
| 检查日期 | {YYYY-MM-DD} |

#### 步骤 1.4：提取设计要求

从被评估项目 README.md 提取设计要求，用于第三阶段对照：

```
被评估项目 README.md 设计要求（来自实际文档）:
1. [分层规则] {提取内容}
2. [鉴权规则] {提取内容}
3. [技术约束] {提取内容，如禁止某类代码}
4. [禁止事项] {提取内容}
5. [优先级定义] {P0/P1 的具体含义}
```

### 3.3 阶段 2：代码审查（双视角）

#### 视角 A：按被评估项目 README.md 设计要求检查（核心）

逐条对照 README.md 中的设计要求，检查代码实现是否满足：

| 被评估项目 README.md 设计要求 | 代码实现 | 是否满足 | 违反定级 |
|----------------|----------|----------|----------|
| {规则1} | {实现情况} | 是/否 | P0/P1/P2/P3 |

**重点检查**：
1. **分层违规**：README.md / AGENTS.md 定义的分层规则是否被遵守
2. **禁止事项**：README.md / AGENTS.md 明确禁止的代码模式是否出现
3. **技术约束**：如"禁止阻塞代码"、"禁止同步调用"是否被违反
4. **鉴权规则**：README.md 描述的认证策略是否被正确实现

#### 视角 B：通用缺陷模式检查（补充）

使用 rules_{语言}.md 进行清单式检查，重点关注跨语言通用缺陷：

| 缺陷类型 | 检查方法 | 定级 |
|----------|----------|------|
| 认证失败路径信任客户端输入 | 找到认证失败的分支，看是否仍使用客户端可控的参数放行 | P0 |
| 认证成功路径信任客户端标识 | 找到认证成功的分支，看用的是服务端生成标识还是客户端传入标识 | P0 |
| 注入漏洞 | 看参数是否直接拼接到 SQL/命令/查询中 | P0 |
| 竞态条件 | 看"先查后改"操作是否有并发保护 | P1 |
| 敏感数据泄露 | 看日志/响应中是否打印/返回敏感信息 | P1/P2 按日志级别 |
| 资源泄漏 | 看流/连接/线程池是否正确关闭 | P1 |
| 错误处理不当 | 看异常是否被静默吞掉 | P1 |
| 配置硬编码 | 看阈值/规则是否支持外部配置 | P2 |

#### 日志级别与敏感信息定级

| 日志级别 | 生产是否输出 | 定级 |
|---------|-------------|------|
| error/warn/info | ✅ 是 | P1 |
| debug | ❌ 否（默认） | P2 |
| trace | ❌ 否 | P3 |

#### 检查输出格式

- `✅ 通过` - 无问题
- `🔴 P0: 问题 - 文件:行号` - 危险，有代码证据
- `🟠 P1: 问题 - 文件:行号` - 告警
- `🟡 P2: 问题 - 文件:行号` - 建议
- `🔵 P3: 问题 - 文件:行号` - 轻微

### 3.4 阶段 3：业务流程走查（发现 P0 的关键）

> 第二阶段发现"代码有没有做某事"，第三阶段验证"做的事对不对"。P0 级问题只有通过走查业务流程才能发现。

#### 步骤 3.1：API 入口走查

对每个 API 端点沿请求链路走一遍：

```
端点: {METHOD} {PATH}
├── 认证: 认证失败/异常后是否放行？认证成功路径用服务端标识还是客户端标识？
├── 鉴权: 无认证路径是否可达受保护资源？权限校验是否完整？
├── 入参: 必填参数是否有校验？
├── 调用链: 请求 → 中间件/拦截器 → 业务逻辑 → 数据层
├── 异常: 每层异常是否正确处理？
└── 响应: 是否泄露敏感信息？
```

#### 步骤 3.2：按设计要求逐条验证

将 README.md 设计要求与代码行为对照：

| README.md 设计要求 | 代码实际行为 | 是否违反 | 定级 |
|----------------|-------------|----------|------|
| {设计要求内容} | {实际代码行为} | 是/否 | P0/P1/P2/P3 |

#### 步骤 3.3：状态流转走查

对有状态的对象：
- 画出状态机
- 验证每个流转的前置条件
- 查找非法跳转、终态可操作、状态竞态窗口

#### 步骤 3.4：并发安全走查

对每个"先查后改"操作：
- 检查查询和更新之间是否有并发保护（乐观锁/悲观锁/唯一约束）
- 检查一次性资源是否有防重机制

#### 步骤 3.5：敏感数据走查

- 密钥是否明文存储在 DB/配置文件
- Token/密码是否在响应中返回
- 日志是否按级别区分敏感信息

#### 定级规则（严格按规则执行）

1. README.md（被评估项目自己的）设计要求标注 P0 的，违反即为 P0
2. AGENTS.md（被评估项目自己的）标注 P0 的规范，违反即为 P0
3. rules 标注 P0 的检查项，违反即为 P0
4. 都没标 P0 但问题严重，按 P1
5. **禁止降级**：发现 P0 不能写成 P1

---

### 3.5 阶段 4：问题合并 + 评分 + 验证 + 报告

#### 步骤 4.1：问题合并去重

- 同一位置同一问题：保留定级高的
- 同一位置不同角度：保留两个
- 第三阶段定级更高：以第三阶段为准

#### 步骤 4.2：扣分规则

| 定级 | 扣分 |
|------|------|
| P0 | -10 |
| P1 | -5 |
| P2 | -3 |
| P3 | -1 |

#### 步骤 4.3：维度扣分上限

各维度扣分上限 = 该维度权重分值。

#### 步骤 4.4：评级标准

| 条件 | 评级 |
|------|------|
| 有任意 P0 | 🔴 不合格 |
| 无 P0，评分 ≥90 | 🟢 优秀 |
| 无 P0，评分 80-89 | 🟡 合格 |
| 无 P0，评分 70-79 | 🟠 待整改 |
| 无 P0，评分 <70 | 🔴 不合格 |

#### 步骤 4.5：结论判定（严格三选一）

| 条件 | 结论 | 禁止 |
|------|------|------|
| 有任意 P0 | **暂停上线** | 禁止写其他 |
| 无 P0 有 P1 | **限期整改后上线** | 禁止写其他 |
| 无 P0 无 P1 | **可上线** | 禁止写其他 |

#### 步骤 4.6：数字交叉验证

**第一轮：问题数量对比**

统计第3部分每个表格实际行数，与第2部分"合计"行对比。

**第二轮：评分验证**

用计算器验证评分：
```bash
node scripts/score_calculator.js '<问题JSON>'
```

**第三轮：结论验证**

- 有 P0 → 暂停上线
- 无 P0 有 P1 → 限期整改后上线
- 无 P0 无 P1 → 可上线

**验证通过才能保存报告。**

#### 步骤 4.7-4.8：生成报告并保存

按 report.md 模板生成报告，包含 6 部分，保存到 `{被评估项目根目录}/质量报告_{项目名}_{日期}.md`

## 4. 评估维度体系

### 4.1 十大评估维度

| 维度 | 权重 | 扣分上限 | 说明 |
|------|------|----------|------|
| 代码规范性 | 5% | 5 | 命名、格式、注释、魔法值 |
| 基础健壮性 | 10% | 10 | 空值、越界、类型转换 |
| **业务逻辑正确性** | **20%** | **20** | 流程、分支、状态、幂等 |
| 整体性能优化 | 10% | 10 | 循环、N+1、批量操作 |
| 并发与线程安全 | 10% | 10 | 锁、线程池、并发控制 |
| 资源管理合理性 | 8% | 8 | 流、连接、内存 |
| 异常处理与容错性 | 10% | 10 | 捕获、熔断、降级 |
| **代码安全合规** | **12%** | **12** | 鉴权、注入、脱敏 |
| 可维护与可扩展性 | 10% | 10 | 耦合、扩展、设计模式 |
| 工程化交付质量 | 5% | 5 | 入参校验、日志、配置 |

**合计**: 100%

### 4.2 定级定义

| 定级 | 说明 | 处理方式 |
|------|------|----------|
| P0 🔴 | 危险缺陷：线上必现故障、数据错乱、高危安全漏洞、权限越权、注入攻击、非法文件上传 | 直接驳回，不予上线 |
| P1 🟠 | 告警缺陷：高概率线上异常、空指针、事务失效、连接泄漏、N+1 查询、并发脏数据、敏感信息泄露 | 限期整改 |
| P2 🟡 | 建议缺陷：代码不规范、逻辑冗余、硬编码、嵌套超标、维护性差、边界场景缺失处理 | 按计划优化 |
| P3 🔵 | 轻微建议：排版、注释、命名细节、工具类优化、日志美化、代码精简 | 可选优化 |

### 4.3 豁免规则

- **历史旧代码**: 长期稳定运行的旧代码可酌情放宽
- **底层框架代码**: 通用底层代码可酌情放宽
- **新增代码**: 100% 严格执行
- **迭代修改代码**: 100% 严格执行

## 5. 语言特定规则

### 5.1 Java / Spring 检查重点

| 检查项 | 定级 | 说明 |
|--------|------|------|
| domain层依赖Spring注解 | P0 | @Service/@Component等禁止出现在domain |
| common层依赖框架 | P0 | 零框架依赖，纯Java+Lombok |
| @Transactional在Controller层 | P0 | 事务必须在Service/Application层 |
| 缓存无容量/淘汰机制 | P0 | 无限新增不清理 |
| Redis连接未关闭 | P0 | 连接泄漏 |

### 5.2 Node.js 检查重点

| 检查项 | 定级 | 说明 |
|--------|------|------|
| 全局错误处理缺失 | P0 | 无统一异常捕获 |
| 使用eval/exec危险方法 | P0 | 代码注入风险 |
| 缓存无容量/淘汰机制 | P0 | 无限新增不清理 |
| Redis连接未关闭 | P0 | 连接泄漏 |
| 未处理的Promise rejection | P1 | 静默失败 |

## 6. 核心组件说明

### 6.1 SKILL.md（主技能文件）

- 版本: 9.0
- 功能: 定义 4 阶段评估流程的执行指南
- 特点: 严格的强制执行规则，禁止跳过任何阶段

### 6.2 report.md（报告模板）

报告模板包含 6 部分：
1. 项目概况
2. 评估结果汇总（维度评分表）
3. 问题详情（P0/P1/P2/P3 分级表格）
4. 修复建议（按优先级分类）
5. 结论（三选一：可上线/限期整改后上线/暂停上线）
6. 检查规则

### 6.3 score_calculator.js（评分计算器）

命令行评分工具，支持：

```bash
# 基本用法
node score_calculator.js '[{"dimension":"代码安全合规","severity":"P0","count":1}]'

# 批量问题
node score_calculator.js '[{"dimension":"基础健壮性","severity":"P1","count":2},{"dimension":"代码规范性","severity":"P2","count":3}]'

# 帮助信息
node score_calculator.js --help
```

输出示例：
```
========== 评分结果 ==========

综合评分: 82/100 🟡 合格
是否允许合入: ❌ 不允许

问题统计:
  P0 致命: 1 个
  P1 严重: 2 个
  P2 一般: 3 个
  P3 优化: 0 个

各维度得分:
  代码规范性: 2/5 (40%)
  基础健壮性: 5/10 (50%)
  ...
```

### 6.4 rules_{语言}.md（检查规则）

每个语言版本的 rules 文件包含：

| 维度 | 检查项数量 | 说明 |
|------|-----------|------|
| 代码规范性 | 21 项 | 命名、格式、注释等 |

## 8. 技术细节

### 8.1 评分公式

```
维度得分 = max(0, 权重分值 - 扣分总和)
综合评分 = Σ(维度得分)
```

**示例**:
- 代码规范性: 5 - 10 = 0 (扣分超出上限)
- 业务逻辑正确性: 20 - 15 = 5
- 综合评分: 0 + 5 + ... = 75 分 (待整改)

### 8.2 加倍规则

同类问题第二次及以上出现，加倍扣分。

### 8.3 问题定级优先级

1. README.md 标注 P0 → P0
2. AGENTS.md 标注 P0 → P0
3. rules 标注 P0 → P0
4. 问题严重但无标注 → P1

### 8.4 禁止降级原则

发现 P0 问题必须定级为 P0，禁止写成 P1。

## 9. 总结

此代码质量评估技能的核心价值在于：

1. **项目定制化评估**: 以被评估项目自己的设计文档为标准，而非通用规范
2. **P0 发现能力**: 通过业务流程走查发现传统清单检查无法发现的深层问题
3. **严格验证机制**: 数字交叉验证确保报告准确性
4. **明确结论**: 三选一结论（可上线/限期整改后上线/暂停上线）避免模糊表述
5. **证据驱动**: 每个问题必须有代码证据（文件:行号），避免无根据的问题报告
