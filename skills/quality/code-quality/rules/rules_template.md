# rules_{language}.md · 规则模板

> 本文件是新语言规则集的写作模板。复制为 `rules_{language}.md`(如 `rules_python.md`),
> 按本模板结构补全后,即可作为该语言的检查规则被 AI 加载。
>
> 参考已有的 `rules_java.md`(302 行)和 `rules_nodejs.md`(269 行)。

---

## 章节结构(必含)

### 1. 命名规范
### 2. 代码组织(分包/分文件/分模块)
### 3. 异常与错误处理
### 4. 日志规范
### 5. 并发与线程安全
### 6. 资源管理(连接/文件/锁)
### 7. 安全(注入/敏感信息/认证)
### 8. 性能(N+1/缓存/批处理)
### 9. 测试(单测覆盖/可测性)
### 10. 框架特性(目标语言主流框架的最佳实践)

---

## 单条规则格式

每条规则必须按以下格式书写,以便 AI 解析与定级:

```markdown
### {RULE-ID} · {规则简述}

- **定级**: P0 / P1 / P2 / P3
- **场景**: 何时该规则触发
- **反例**:
  ```{language}
  // 反例代码
  ```
- **正例**:
  ```{language}
  // 正例代码
  ```
- **检查方式**: grep / AST 检查 / 编译期检查 / 运行时检查 ...
- **修复建议**: 一句话讲清怎么改
```

---

## 定级原则(参考 rules.md)

| 级别 | 适用情形 |
|---|---|
| P0 🔴 | 阻断上线 — 安全漏洞 / 数据丢失 / 服务不可用 |
| P1 🟠 | 高优修复 — 已上线但需在下一个迭代解决 |
| P2 🟡 | 改进项 — 影响可维护性但不影响功能 |
| P3 🔵 | 参考建议 — 风格统一 / 微小优化 |

---

## 质量门槛

- 规则数 ≥ 150 条
- 章节覆盖 1~10 全部
- P0~P3 四级均要有(避免全是 P2 的"无效规则集")
- 主流框架 Top 3 必须覆盖
